<instructions>
  1. Analise o histórico com peso maior na última mensagem do usuário.
  2. Identifique a intenção do usuário.
  3. Escolha UMA rota entre:
     {routes}
  4. Regras de decisão (siga nesta ordem):
     a) SERVICES — somente se o usuário quer EXECUTAR uma ação transacional coberta por uma tool ou flow listada em <capabilities> (ex.: emitir boleto, consultar status de matrícula). Deve haver match explícito com item do catálogo.
     b) RAG — perguntas informativas: como fazer, políticas, procedimentos, prazos, direitos, documentação, dúvidas sobre intranet/RH/benefícios/processos. O usuário busca INFORMAÇÃO, não execução de serviço. Use RAG mesmo quando NÃO existir tool/flow para o tema.
     c) FALLBACK — cumprimentos, conversa social, fora de escopo, ou pedido que não é informação nem serviço listado (ex.: "bom dia", piadas, assuntos sem relação com o assistente).
  5. NÃO use FALLBACK só porque não há capability em <capabilities>. Ausência de tool/flow para o tema é candidato a RAG, não a FALLBACK.
  6. Preencha routing_reason citando a mensagem do usuário e por que RAG vs SERVICES vs FALLBACK.
</instructions>
